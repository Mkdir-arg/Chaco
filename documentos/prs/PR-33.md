## Resumen
Este PR rediseña el stack local de Docker para reducir el tiempo hasta entorno usable y bajar la fragilidad del arranque diario.

El entorno recomendado pasa a levantarse con un solo comando:

```bash
docker compose up --build
```

Sobre un stack mínimo de `app + mysql + redis`, sin `nginx` ni separación artificial entre HTTP y WebSocket.

## Problema
El stack híbrido local anterior degradaba mucho la DX porque:

- reinstalaba dependencias en cada arranque
- mezclaba runtime y bootstrap pesado en el mismo flujo
- ejecutaba comandos no esenciales en cada restart
- dependía de `nginx + gunicorn + daphne` para un entorno local donde eso no aportaba valor real
- tenía un bootstrap roto en `crear_programas` por uso del campo legacy `activo`

## Qué cambia
### Stack local
- se agrega `docker-compose.yml` como entrypoint local por defecto
- el stack local recomendado queda reducido a:
  - `app`
  - `mysql`
  - `redis`

### Runtime de aplicación
- `app` sirve el backend local en un solo puerto (`8000`)
- HTTP y WebSocket comparten un único proceso ASGI
- el runtime local por defecto pasa a `python manage.py runserver 0.0.0.0:8000`
- se mantiene fallback explícito con `APP_RUNTIME=daphne`

### Bootstrap
- el entrypoint local ahora hace solo:
  - espera de DB
  - `migrate`
  - `crear_superadmin`
  - `setup_grupos`
  - `crear_programas`
  - arranque del runtime
- salen del startup diario:
  - `pip install -r requirements.txt`
  - `sleep`
  - `collectstatic`
  - `load_initial_data`
  - `setup_system`

### Configuración
- se agrega `.env.local.example`
- se documenta el flujo local y el bootstrap opcional
- `config/settings.py` suma `app` a los hosts internos permitidos

### Fix incluido
- `legajos/management/commands/crear_programas.py` deja de usar `activo` y pasa a crear/actualizar programas con `estado` y `naturaleza` del modelo actual

## Riesgos / tradeoffs
- el entorno local prioriza DX sobre paridad estructural con el despliegue legado
- el runtime local default usa `runserver`; si alguna pantalla con Channels necesitara fallback, sigue disponible `APP_RUNTIME=daphne`
- no se hizo en esta rama una prueba funcional end-to-end de un flujo WebSocket específico de negocio; sí quedó validado el runtime ASGI local y el autoreload

## Manual test sugerido
1. Ejecutar `docker compose up --build`.
2. Verificar `http://localhost:8000/health/`.
3. Iniciar sesión con `admin / mkdir123`.
4. Editar un archivo Python y confirmar `changed, reloading` en `docker compose logs app -f`.
5. Reiniciar con `docker compose down` y `docker compose up` y validar que ya no corran seeds pesados en cada restart.
6. Opcional: levantar con `APP_RUNTIME=daphne docker compose up --build` para verificar fallback.
