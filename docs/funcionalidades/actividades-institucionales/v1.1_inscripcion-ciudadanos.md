## Descripcion

Permite inscribir ciudadanos a actividades institucionales desde el backoffice (operadores) y desde el portal ciudadano. Valida tipo de acceso (LIBRE/REQUIERE_PROGRAMA), unicidad de inscripción activa, cupo disponible y genera un código de inscripción único de 8 caracteres alfanuméricos. El modelo soporta historial de reinscripciones: un ciudadano dado de ABANDONADO o FINALIZADO puede reinscribirse.

## User Story

```
Como operador backoffice o ciudadano autenticado en el portal
quiero inscribir un ciudadano a una actividad institucional
para registrar su participación con código de confirmación,
respetando las reglas de tipo de acceso y cupo.
```

## Criterios de aceptacion

- [x] Inscripción desde backoffice por operador autenticado con formulario de búsqueda por DNI
- [x] Inscripción desde portal por ciudadano (solo actividades LIBRE visibles)
- [x] Validación de tipo de acceso reutilizando `validar_acceso_actividad` existente
- [x] Validación de cupo (cupo_ciudadanos >= 1 como límite; 0 = ilimitado)
- [x] Bloqueo si ya tiene inscripción INSCRITO o ACTIVO en la misma actividad
- [x] Generación de `codigo_inscripcion` (8 caracteres alfanuméricos únicos, generado con `secrets`)
- [x] Registro de `inscrito_por` para trazabilidad
- [x] El código es visible en la nómina del backoffice y en el portal ciudadano
- [x] Historial de reinscripciones permitido (un ciudadano puede reinscribirse tras ABANDONADO/FINALIZADO)
- [x] Bug corregido: badge de cupo en actividad_detail.html ahora trata cupo=0 como "Sin límite"

## Decisiones tecnicas

- **Constraint de unicidad activa vía service, no DB**: MySQL 8.0 no soporta índices parciales condicionales. La invariante "solo un registro activo por (ciudadano, actividad)" se garantiza via `@transaction.atomic + select_for_update()` en `inscribir_ciudadano_a_actividad`.
- **Código generado con `secrets.choice`**: espacio de 36^8 ≈ 2.8B combinaciones; colisión prácticamente imposible con el volumen esperado. Loop de retry con máx. 10 intentos.
- **`aceptar_derivacion` delegado al service central**: se eliminó el `get_or_create` inline que no consideraba el estado ni la semántica de reinscripción.

## Limitaciones conocidas / deuda tecnica

- La vista del portal no incluye paginación en el listado de actividades disponibles (aceptable para el volumen actual).
- No hay notificación por email al inscribirse (queda para versión futura).
- La solapa "Cursos y Actividades" del hub ciudadano (US-009) aún no existe; la URL `/mi-perfil/actividades/` es accesible pero no aparece en el menú lateral hasta que se implemente US-009.
