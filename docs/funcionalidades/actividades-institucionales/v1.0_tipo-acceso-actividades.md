## Descripcion

Permite configurar el tipo de acceso de una actividad institucional (`PlanFortalecimiento`): libre (sin requisito) o restringida (requiere inscripción activa en un programa social). Es prerequisito directo de US-022 (inscripción a actividades), que consume la función de validación `validar_acceso_actividad`.

## User Story

```
Como administrador quiero configurar el tipo de acceso de una actividad
(LIBRE o REQUIERE_PROGRAMA) para controlar qué ciudadanos pueden inscribirse,
asegurando que las actividades restringidas solo admitan ciudadanos activos
en el programa correspondiente.
```

## Criterios de aceptacion

- [x] Todas las actividades existentes quedan con `tipo_acceso = LIBRE` al aplicar la migración
- [x] El formulario de creación muestra el campo `tipo_acceso` con Alpine.js show/hide para `programa_requerido`
- [x] El formulario de edición tiene la misma lógica condicional
- [x] Si `tipo_acceso = REQUIERE_PROGRAMA` y no se selecciona programa → error de validación
- [x] Si `tipo_acceso = LIBRE` → `programa_requerido` se limpia a null automáticamente al guardar
- [x] `validar_acceso_actividad(actividad, ciudadano)` retorna `(True,'')` para LIBRE y `(False, mensaje)` si el ciudadano no cumple el requisito
- [x] El selector de portal `get_actividades_accesibles(ciudadano)` filtra correctamente usando 2 queries (anti N+1)

## Dependencias

- Depende de: US-006 no directamente; `Programa` de `legajos.models_programas`; `InscripcionPrograma` para la validación
- Usada por: US-022 (inscripción a actividades) — consume `validar_acceso_actividad`

## Decisiones tecnicas

- **FK lazy `'legajos.Programa'`:** Se usa string reference para evitar import circular entre `models.py` y `models_programas.py`. Django resuelve en el registry de apps.
- **Alpine.js + Select2:** el `x-show` se aplica al `<div>` contenedor del campo, no al `<select>` directamente, para que Select2 (que reemplaza el elemento nativo) respete el show/hide.
- **Anti-N+1 en el selector:** pre-carga de IDs de programas activos del ciudadano en un `set` + filtro Q combinado. Siempre 2 queries independientemente del volumen.
- **Actividad mal configurada no bloquea:** si `REQUIERE_PROGRAMA` pero `programa_requerido` es null, la validación deja pasar (graceful degradation — evita bloquear ciudadanos por errores de configuración).

## Limitaciones conocidas / deuda tecnica

- El botón de acceso en el portal (vista que liste actividades para el ciudadano) no está implementado en esta US. El selector `get_actividades_accesibles` es la infraestructura; la vista y template de portal quedan para US-022/US-015.
- No hay guard de permisos explícito en los formularios (solo `@login_required` heredado de las vistas). Si se implementa `programaConfigurar` como rol requerido, deberá agregarse en las views de `configuracion/`.
