# Hub del Ciudadano — Solapas Dinámicas y Badge Behavior — v1.1

> Version: v1.1
> Fecha: 2026-03-19
> Tipo: EVOLUTIVO
> App/Modulo: `legajos`

---

## Descripcion

Expande el Hub del Ciudadano (v1.0) incorporando 7 solapas estáticas nuevas (Turnos, Instituciones, Conversaciones, Derivaciones, Alertas, Línea de tiempo, Red Familiar, Archivos) y un sistema de badges dinámicos sobre las solapas. Los badges muestran conteos numéricos para ítems de atención urgente (alertas, derivaciones, turnos próximos, mensajes no leídos) y un punto de presencia para Legajos. Las solapas de programas mantienen el comportamiento dinámico por inscripción activa, y su ID se genera siempre a través de `SolapasService._normalizar_tipo_programa` para garantizar consistencia entre el botón y el panel.

## User Story

```
Como operador/profesional del backoffice quiero ver en el hub del ciudadano
todas las solapas de información (turnos, instituciones, conversaciones,
derivaciones, alertas, línea de tiempo) con badges que indican ítems de
atención, para priorizar la acción sin tener que navegar entre módulos.
```

## Criterios de aceptacion

- [x] Las solapas Turnos, Instituciones, Conversaciones, Derivaciones, Alertas y Línea de tiempo están visibles para todos los ciudadanos
- [x] La solapa Alertas muestra badge rojo con el conteo de alertas activas
- [x] La solapa Derivaciones muestra badge naranja con el conteo de derivaciones pendientes
- [x] La solapa Turnos muestra badge azul con el conteo de turnos en los próximos 7 días
- [x] La solapa Conversaciones muestra badge violeta con mensajes no leídos del ciudadano
- [x] La solapa Legajos muestra punto gris si tiene al menos 1 legajo
- [x] Los tabs vacíos muestran empty state descriptivo
- [x] Los tabs de programas dinámicos usan el ID normalizado de `SolapasService`
- [x] Las confirmaciones de eliminación usan SweetAlert2

## Decisiones tecnicas

**Inline styles para colores de badge:** Los valores hex de color se pasan como contexto Python (`color_hex`). Tailwind purga clases dinámicas como `bg-{{ color }}-500`, por lo que se usan inline styles `style="background-color: {{ badge.color_hex }};"` con valores hardcodeados en `obtener_badges_ciudadano`.

**Loop de tab con `solapas` en vez de `programas_activos`:** El panel del tab de ACOMPANAMIENTO_SEDRONAR ahora itera `solapas` filtrando por `not solapa.estatica and 'ACOMPANAMIENTO_SEDRONAR' in solapa.id`. Esto garantiza que el ID del panel usa `solapa.id` (normalizado por `_normalizar_tipo_programa`) y siempre coincide con el ID que genera el botón de tab.

**Copia de dicts en SOLAPAS_ESTATICAS:** `dict(s)` en `obtener_solapas_ciudadano` evita que la lista de clase se mute entre requests. El badge se inyecta con `{**s, 'badge': ...}` sin modificar el dict original.

**Alertas generadas on-the-fly:** `AlertasService.generar_alertas_ciudadano(ciudadano.pk)` se llama en `build_ciudadano_detail_context` antes de consultar las alertas, en un bloque `try/except` para no romper la carga del hub si falla.

## Limitaciones conocidas / deuda tecnica

- `TurnoCiudadano` y `Mensaje` se importan lazily desde `legajos`, creando dependencia implícita entre apps. Documentado como riesgo arquitectónico en `decisions.md`.
- La solapa "Cursos y Actividades" está visible pero no tiene datos propios — requiere US de Actividades Institucionales.
- La solapa "Red Familiar" tiene panel vacío — requiere integración con el modelo de contactos.
- La solapa "Archivos" tiene lógica de carga AJAX ya existente, no agregada en esta versión.
- El badge de `linea_tiempo` no está implementado (no hay métrica clara de "novedad").

## Cambios respecto a version anterior (v1.0)

- v1.0 tenía 4 solapas estáticas (Resumen, Cursos y Actividades, Línea de tiempo, Archivos) + solapas dinámicas por programa.
- v1.1 expande a 11 solapas estáticas agregando Legajos, Turnos, Instituciones, Conversaciones, Derivaciones, Alertas, Red Familiar.
- v1.1 agrega sistema de badges con 5 tipos (alertas rojo, derivaciones naranja, turnos azul, conversaciones violeta, legajos punto gris).
- v1.1 corrige el ID del panel de tab de programas dinámicos para usar siempre el tipo normalizado del servicio.
- v1.1 migra `confirm()` nativo a SweetAlert2 en `eliminarArchivo` y `eliminarVinculo`.
