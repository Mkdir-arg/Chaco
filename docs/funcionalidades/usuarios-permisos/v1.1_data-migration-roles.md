# Usuarios y Permisos — v1.1

> Version: v1.1
> Fecha: 2026-03-15
> Tipo: EVOLUTIVO
> App/Modulo: `users`, `turnos`

---

## Descripcion

Unificación de todos los grupos Django del sistema con los nombres definitivos acordados en la sesión de /definir roles. Incluye la creación idempotente de 15 roles operativos + 5 grupos especiales via management command, y la migración de 4 grupos legacy con nombres viejos. Cierra la inconsistencia entre el código de permisos en `turnos/mixins.py` y el nombre de grupo real en la base de datos.

## User Story

```
Como administrador del sistema quiero que todos los roles del backoffice existan
como grupos Django con sus nombres definitivos acordados, y que los grupos con
nombres viejos sean migrados automáticamente, para que cualquier feature de
permisos pueda implementarse sobre una base estable y sin inconsistencias.
```

## Criterios de aceptacion

- [x] Al ejecutar `python manage.py setup_grupos`, existen los 15 roles operativos con nombres definitivos
- [x] Existen los 5 grupos especiales: `Ciudadanos`, `EncargadoInstitucion`, `AdministrativoInstitucion`, `ProfesorInstitucion`, `Responsable`
- [x] El command es idempotente — ejecutarlo múltiples veces no genera duplicados
- [x] `configurarSecretaria` → `secretariaConfigurar` (usuarios preservados)
- [x] `ConfiguracionPrograma` → `programaConfigurar` (usuarios preservados)
- [x] `Administradores de Turnos` → `turnoConfigurar` (usuarios preservados)
- [x] `EncargadoDispositivo` → `EncargadoInstitucion` (usuarios preservados)
- [x] `turnos/mixins.py` usa `'turnoConfigurar'` en ambos puntos de control de acceso
- [x] Ninguna referencia al nombre viejo `'Administradores de Turnos'` en código de producción

## Dependencias

- Depende de: ninguna (es el prerequisito base)
- Desbloquea: US-010, US-012, US-018, corrección de errores de permisos en turnos

## Decisiones tecnicas

El management command `setup_grupos` ya existía y estaba sustancialmente implementado. La decisión de usar un command (en lugar de una data migration de Django) es correcta porque los grupos de `auth` no son parte del dominio de la app — son configuración del sistema que puede necesitar re-ejecutarse en cualquier entorno (dev, staging, prod).

El cuarto renombre (`EncargadoDispositivo` → `EncargadoInstitucion`) no estaba documentado en el requerimiento original pero sí estaba en el command. Se incorpora a la documentación en esta versión.

## Limitaciones conocidas / deuda tecnica

- Las vistas operativas de turnos (agenda, aprobar, rechazar, cancelar) aún no requieren `turnoOperar` — siguen usando `operador_required` genérico. Esto se corrige inline cuando se implemente la US correspondiente al módulo de turnos.
- El error `2026-03-11_permisos-turnooperar-no-aplicado.md` permanece ABIERTO — no era alcance de esta US.
