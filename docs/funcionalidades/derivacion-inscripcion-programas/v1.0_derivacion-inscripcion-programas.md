# Derivación e Inscripción de Ciudadanos a Programas — v1.0

> Version: v1.0
> Fecha: 2026-03-15
> Tipo: FEATURE
> App/Modulo: `legajos`

---

## Descripcion

Implementa el flujo completo de derivación e inscripción directa de ciudadanos a programas sociales. Un operador puede derivar un ciudadano a un programa (creando una DerivacionCiudadano en estado PENDIENTE) o inscribirlo directamente (si tiene rol programaOperar). El operador destino acepta o rechaza desde la bandeja del programa; al aceptar se crea la InscripcionPrograma y el FlowRuntime inicia el flujo automáticamente via signal. Reemplaza el flujo previo basado en DerivacionPrograma, que queda como tabla legacy para Ñachec hasta su migración.

## User Story

```
Como operador del backoffice quiero derivar o inscribir directamente a un ciudadano
en un programa para iniciar su proceso de admisión formal a través del flujo
configurado del programa
```

## Criterios de aceptacion

- [x] Desde la ficha del ciudadano se puede crear una DerivacionCiudadano a un programa destino (InstitucionPrograma)
- [x] Si ya existe derivación PENDIENTE al mismo programa → bloquear con aviso
- [x] Si el ciudadano ya está ACTIVO en ese programa → bloquear con aviso
- [x] Si la derivación previa fue RECHAZADA → permitir una nueva sin restricción
- [x] Solo usuarios con grupo `programaOperar` pueden hacer inscripción directa (`tipo_inicio=INSCRIPCION_DIRECTA`)
- [x] La bandeja del dashboard del programa muestra DerivacionCiudadano PENDIENTES con stats (pendientes/aceptadas/rechazadas)
- [x] Al aceptar: `estado=ACEPTADA` + crea `InscripcionPrograma` + FlowRuntime inicia via signal
- [x] Si existe inscripción previa cerrada → se reactiva en lugar de crear duplicado (respeta unique_together)
- [x] Al rechazar: campo de motivo obligatorio, `estado=RECHAZADA`, NO crea InscripcionPrograma
- [x] `puede_operar_programa` corregido para verificar grupo `programaOperar`, responsable local y coordinadores activos
- [x] URL names `derivacion_ciudadano_aceptar` y `derivacion_ciudadano_rechazar` para el nuevo flujo
- [x] URLs legacy `derivacion_aceptar` y `derivacion_rechazar` preservadas para Ñachec
- [x] Todos los badges del template usan `badge-nodo` del design system

## Dependencias

- Depende de: US-021 (DerivacionCiudadano implementado), US-006 (FlowRuntime via signal en post_save de InscripcionPrograma), US-011 (grupos Django con nombres definitivos)
- Usada por: US-015 (portal ciudadano — sección Programas), US-017 (baja de ciudadano del programa)

## Decisiones tecnicas

- **Reemplazo directo de DerivacionPrograma:** en lugar de construir el nuevo flujo en paralelo, se migra la UI activa a DerivacionCiudadano (Opción A). `DerivacionPrograma` queda como tabla legacy sin UI, igual que se hizo con `DerivacionInstitucional` en US-021.
- **URLs separadas para Ñachec:** se usaron URL names nuevos (`derivacion_ciudadano_*`) para el nuevo flujo, preservando los legacy (`derivacion_aceptar/rechazar`) para el bloque Ñachec que aún usa `DerivacionPrograma`.
- **FlowRuntime via signal:** el service no invoca `FlowRuntime.iniciar()` directamente. El signal `iniciar_flujo_inscripcion` en `legajos/signals/programas.py` ya lo hace en `post_save` de `InscripcionPrograma`, evitando doble llamada.
- **Reactivación de inscripción cerrada:** ante el `unique_together` de `InscripcionPrograma`, si existe un registro previo cerrado se reactiva con `estado=PENDIENTE` en lugar de intentar crear un duplicado que produciría `IntegrityError`.
- **Corrección de permisos:** `puede_operar_programa` y similares ahora verifican el grupo Django `programaOperar` como primer check tras `is_superuser`, desbloqueando el flujo para todos los usuarios con ese rol. La granularidad fina por `InstitucionPrograma` específica queda como deuda técnica futura (requiere implementar `UsuarioInstitucionPrograma`).

## Limitaciones conocidas / deuda tecnica

- El bloque Ñachec en `ProgramaDetailView` y `programa_nachec_detail.html` sigue usando `DerivacionPrograma`. La migración completa de Ñachec al nuevo modelo queda para cuando se migre ese dominio.
- `puede_operar_programa` permite operar en todos los programas a cualquier usuario con `programaOperar` (sin restricción por InstitucionPrograma específico). La granularidad fina requiere implementar `UsuarioInstitucionPrograma`.
- La UI para derivar desde el portal ciudadano es parte de US-015 (fuera de este alcance).
- No hay notificaciones al operador destino cuando llega una derivación (queda para US-016).
