## Descripcion

Aplicación del sistema de roles definido en US-011 al módulo de instituciones. Reemplaza el control de acceso ad-hoc (verificaciones manuales de `is_superuser`, filtros por institución de usuario) por el patrón unificado `GroupRequiredMixin` / `@group_required`. Dos roles operativos: `institucionVer` (lectura) e `institucionAdministrar` (escritura completa).

## User Story

```
Como administrador quiero que las vistas de instituciones estén protegidas por los roles `institucionVer` e `institucionAdministrar` para controlar el acceso al módulo
```

## Criterios de aceptacion

- [x] Usuarios sin ninguno de los dos roles son redirigidos con mensaje de error
- [x] `institucionVer` accede a listado y detalle pero no puede crear/editar/eliminar
- [x] `institucionAdministrar` tiene acceso completo a todas las instituciones sin filtro por usuario
- [x] Superusuarios no están bloqueados por ningún grupo
- [x] Usuarios no autenticados son redirigidos al login

## Decisiones tecnicas

- **`GroupRequiredMixin` siempre con `LoginRequiredMixin`**: el mixin tiene guard propio para usuarios no autenticados, pero la convención en CBVs es usarlos juntos para claridad.
- **`institucionAdministrar` ve todas las instituciones**: a diferencia del comportamiento anterior donde el usuario solo veía su institución asignada, el rol de administrador tiene acceso global. Los encargados de institución tendrán su propio panel (US-019).
- **Redirect a `configuracion:dispositivos`**: era el destino más razonable disponible cuando el `success_url` original (`configuracion:instituciones`) no existía.

## Cambios respecto a version anterior

La v1.0 tenía control de acceso manual en cada vista (verificaciones `is_superuser`, `get_queryset` filtrando por `usuario`). Esta versión centraliza el control en `GroupRequiredMixin` y el decorador extendido `group_required`, consistente con el patrón del resto del sistema. El bug de `success_url` (500 al crear institución) queda corregido.
