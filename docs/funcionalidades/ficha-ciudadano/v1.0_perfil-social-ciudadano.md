## Descripcion

Extiende la ficha del ciudadano con datos del perfil social completo: foto, situación habitacional, situación laboral/económica, nivel educativo, cobertura médica, medicación habitual, documentación migratoria y observaciones libres. Centraliza toda la información social en un único perfil accesible desde el backoffice. Los campos de salud y documentación migratoria quedan restringidos a usuarios con el grupo `ciudadanoSensible` o superusuarios, implementando una protección en tres capas: formulario, vista y template.

## User Story

```
Como operador quiero que la ficha del ciudadano incluya situación habitacional,
laboral, educativa, médica, documentación migratoria, notas y foto para tener
toda la información social centralizada.
```

## Criterios de aceptacion

- [x] El operador puede subir y ver la foto del ciudadano desde la pantalla de edición
- [x] La foto se valida (máximo 5 MB) antes de persistir
- [x] El operador puede registrar tipo de vivienda, tenencia y condiciones habitacionales
- [x] El operador puede registrar situación laboral, ingreso estimado y obra social
- [x] El operador puede registrar nivel educativo
- [x] Los campos sensibles (cobertura médica, medicación habitual, estado migratorio) solo son visibles y editables por usuarios en el grupo `ciudadanoSensible` o superusuarios
- [x] El operador puede registrar datos de documentación (DNI físico, estado RENAPER, estado migratorio)
- [x] El operador puede ingresar observaciones libres
- [x] La vista de detalle muestra la sección social solo cuando hay datos cargados (`{% if tiene_social %}`)
- [x] La vista de detalle muestra la sección sensible solo cuando el usuario tiene el permiso correspondiente (`{% if puede_ver_sensible %}`)
- [x] El formulario de edición usa `enctype="multipart/form-data"` para soportar la carga de foto
- [x] Al guardar, el caché del ciudadano se invalida correctamente

## Decisiones tecnicas

**Control de acceso en tres capas para campos sensibles:**
La protección de los campos `cobertura_medica`, `medicacion_habitual` y `estado_migratorio` se implementó en tres capas independientes para garantizar que ningún punto de entrada los exponga sin autorización:
1. **Form layer:** `__init__` solo inyecta los fields sensibles cuando `puede_ver_sensible=True`; `save()` los persiste solo si el flag es verdadero.
2. **View layer:** calcula `puede_ver_sensible = user.is_superuser or user.groups.filter(name='ciudadanoSensible').exists()` y lo pasa al form y al contexto.
3. **Template layer:** sección sensible envuelta en `{% if puede_ver_sensible %}`.

**`puede_ver_sensible` como flag de contexto en lugar de permiso Django:**
Se decidió usar un flag calculado en la view en lugar de un `permission_required` de Django, porque el acceso no bloquea la vista completa — solo oculta/muestra un subconjunto de campos dentro del mismo formulario y detalle.

**`ImageField` con validación de peso en `clean_foto`:**
La validación de 5 MB máximo se realiza en el form (`clean_foto`) y no en el modelo, siguiendo el patrón establecido en el proyecto de mantener reglas de UI/UX en forms y reglas de negocio estructurales en models.

**Indexes adicionales:**
Se agregaron 4 índices para campos que se proyectan como filtros en el hub ciudadano (US-009): `situacion_laboral`, `nivel_educativo`, `estado_migratorio`, `tipo_vivienda`.

## Limitaciones conocidas / deuda tecnica

- La foto no tiene resize/thumbnail automático — si el usuario sube una imagen grande, se almacena en su tamaño original. Se puede agregar procesamiento con `Pillow` en una versión futura.
- El campo `ingreso_estimado` es de texto libre — en una versión futura podría convertirse a `DecimalField` para permitir análisis estadístico.
- La pregunta "¿La foto requiere `ciudadanoSensible` o la ve cualquiera con `ciudadanoVer`?" permanece abierta en `contexto-funcional.md`. En esta implementación la foto NO es sensible y la puede ver cualquier operador con acceso a la ficha.
