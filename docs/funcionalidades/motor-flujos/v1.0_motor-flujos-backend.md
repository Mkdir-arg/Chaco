# Motor de Flujos — v1.0

> Version: v1.0
> Fecha: 2026-03-15
> Tipo: NUEVO
> App/Modulo: `flujos`

---

## Descripcion

App `flujos/` que provee un motor de ejecución de flujos configurable para programas sociales. Permite definir flujos de atención mediante un JSON con nodos y transiciones, publicar versiones de esos flujos, e iniciar/avanzar instancias de ejecución asociadas a inscripciones de ciudadanos. Sienta las bases para que US-007 (editor visual React Flow) pueda guardar y publicar definiciones vía API.

## User Story

```
Como desarrollador quiero un motor de flujos backend (app flujos/) con modelos, runtime y tipos de nodo
para que los programas puedan tener flujos configurables que guíen la atención de un ciudadano
desde su inscripción hasta el cierre del caso.
```

## Criterios de aceptacion

- [x] `Flujo` (OneToOne con Programa), `VersionFlujo` (JSON + estado BORRADOR/PUBLICADA/ARCHIVADA), `InstanciaFlujo` (OneToOne con InscripcionPrograma), `InstanciaLog` (registro inmutable de transiciones)
- [x] `VersionFlujo.numero_version` auto-incremental por flujo con SELECT FOR UPDATE
- [x] `FlowRuntime.iniciar(inscripcion)` — crea instancia en el nodo inicio
- [x] `FlowRuntime.avanzar(instancia, datos, usuario)` — evalúa condiciones y avanza al siguiente nodo
- [x] Evaluador de condiciones: `==`, `!=`, `>`, `>=`, `<`, `<=`, `in`; `null` = transición libre
- [x] Notificación best-effort (logger.info) al llegar a nodo `accion_humana`
- [x] Nodos temporales (`espera`) solo se registran, sin procesamiento automático
- [x] `GET/POST /api/flujos/<programa_id>/definicion/` — leer y guardar definición
- [x] `POST /api/flujos/<programa_id>/publicar/` — publicar versión borrador
- [x] `GET /api/flujos/instancia/<instancia_id>/` — estado de instancia con nodo completo
- [x] `Programa.flujo_activo` property — devuelve VersionFlujo PUBLICADA más reciente o None
- [x] Validación `BORRADOR → ACTIVO` en programa ahora requiere flujo publicado (además de naturaleza)
- [x] Signal `iniciar_flujo_inscripcion` — dispara el runtime al crear una InscripcionPrograma

## Decisiones tecnicas

- **Sin DRF**: la API de flujos usa FBVs con `JsonResponse`. El proyecto ya tiene DRF para `legajos` y `core`, pero los endpoints del motor son sencillos y no requieren serializers. Mantiene consistencia con los endpoints AJAX existentes en `core`.
- **`numero_version` con SELECT FOR UPDATE**: MySQL no soporta AUTO_INCREMENT con scope parcial fuera de MyISAM. El auto-incremento por flujo se implementa en `save()` con `select_for_update()` dentro de `transaction.atomic()`. El `UniqueConstraint(flujo, numero_version)` actúa como red de seguridad.
- **Unicidad de versión PUBLICADA enforced en aplicación**: MySQL 8.0 no soporta partial unique indexes. La regla "solo una versión PUBLICADA por flujo" se enforcea en `api_publicar` dentro de `transaction.atomic()` archivando las anteriores antes de publicar la nueva.
- **Import diferido en signal**: `from flujos.runtime import FlowRuntime` se hace dentro del body de la función signal para evitar cualquier riesgo de import circular en el startup de Django.
- **Notificaciones best-effort**: las notificaciones de nodo `accion_humana` son solo `logger.info`. Celery/WebSocket es deuda futura (ver Limitaciones).

## Limitaciones conocidas / deuda tecnica

- Las notificaciones de nodo `accion_humana` son solo logging. El canal real (WebSocket, email, notificación en backoffice) requiere integración con Celery o Channels — deuda futura.
- No hay endpoint para `avanzar()` desde el backoffice. Eso corresponde a US-012 (inscripción/derivación) que implementará la UI de avance de nodo.
- Los nodos `espera` no tienen temporizador automático — se procesan solo bajo demanda. Timers automáticos requieren Celery Beat.
- No hay validación de que el JSON de definición sea un grafo conexo (que todos los nodos sean alcanzables desde el inicio). El editor visual (US-007) será responsable de validar la estructura del grafo en el frontend.
