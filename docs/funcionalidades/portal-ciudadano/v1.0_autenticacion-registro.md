## Descripcion

Implementacion de la capa de autenticacion y registro del Portal Ciudadano. Permite que un ciudadano cree una cuenta de acceso al portal asociandola a su legajo existente (o creando un legajo nuevo), inicie sesion, y acceda a su perfil. El portal esta completamente separado del backoffice del sistema: los ciudadanos nunca acceden al panel de operadores.

Esta es la Iteracion 1 de 6 del Portal Ciudadano.

## User Story

```
Como ciudadano quiero registrarme y acceder al portal del municipio para consultar el estado de mis tramites y comunicarme con el equipo sin necesidad de asistir presencialmente
```

## Criterios de aceptacion

- [x] El ciudadano puede registrarse con email y password
- [x] El flujo de registro detecta si ya existe un legajo con ese DNI y lo vincula
- [x] Si el ciudadano ya tenia cuenta, se le informa en lugar de crear un duplicado
- [x] El login tiene rate limiting: 5 intentos fallidos bloquean la IP por 5 minutos
- [x] El ciudadano autenticado que intenta acceder al backoffice es redirigido a su perfil
- [x] El ciudadano puede ver su perfil con datos del legajo vinculado
- [x] El ciudadano puede iniciar un flujo de reset de password via email
- [x] El Step 2 del registro no expone nombre ni apellido del legajo (privacidad)
- [x] Race condition protegida: re-verificacion de `usuario_id` antes de asignar en Step 2

## Decisiones tecnicas

**Separacion absoluta backoffice/portal via middleware:**
El `PortalCiudadanoMiddleware` intercepta cada request. Si el usuario autenticado pertenece al grupo `Ciudadanos` y la URL no es `/portal/` ni `/accounts/`, lo redirige al perfil ciudadano. Esto hace que la separacion sea imposible de bypassear por configuracion de permisos olvidada. Alternativa descartada: solo usar `@login_required` con grupos, porque requiere que cada vista recuerde verificar el grupo тАФ demasiado fr├бgil.

**Rate limiting por IP via cache Django (sin libreria externa):**
5 intentos fallidos de login bloquean la IP por 5 minutos usando `django.core.cache`. Se descarto `django-ratelimit` y `django-axes` para no agregar dependencias. El cache en produccion es Redis (ya configurado para channels), por lo que el bloqueo es efectivo incluso con multiples workers.

**No exponer nombre del legajo en Step 2 del registro:**
Cuando el DNI ingresado coincide con un legajo existente sin cuenta, el Step 2 solo confirma que "existe un legajo con ese DNI" sin mostrar nombre ni apellido. Previene enumeracion de personas via el formulario publico.

**Race condition en asignacion de cuenta a legajo:**
Antes de asignar `legajo.usuario = nuevo_user` en el Step 2, se re-verifica `legajo.usuario_id` en BD (no desde el objeto en memoria) para evitar que dos registros simultaneos vinculen el mismo legajo a dos cuentas distintas.
