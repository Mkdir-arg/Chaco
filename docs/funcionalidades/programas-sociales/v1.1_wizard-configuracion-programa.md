## Descripcion

Wizard de 4 pasos para crear y editar programas sociales, accesible desde el módulo de Configuración. Reemplaza la creación ad-hoc de programas via admin de Django. El model `Programa` incorpora naturaleza del programa (un solo acto / persistente), estado con ciclo de vida formal (BORRADOR → ACTIVO → SUSPENDIDO / INACTIVO), capacidades activables (turnos, cupo, lista de espera) y jerarquía organizacional completa (Secretaría → Subsecretaría → Programa).

## User Story

```
Como usuario con rol programaConfigurar quiero crear y editar un programa social
mediante un wizard de configuración por pasos para dar de alta programas
correctamente tipificados y listos para operar.
```

## Criterios de aceptacion

- [x] Solo `programaConfigurar` accede al wizard — redirige con mensaje si no tiene el grupo
- [x] El listado es visible para cualquier usuario autenticado del backoffice
- [x] Paso 1: nombre (único), código (único), descripción, Secretaría y Subsecretaría (filtrada por AJAX)
- [x] Paso 2: naturaleza obligatoria — `UN_SOLO_ACTO` o `PERSISTENTE` con selección visual
- [x] Paso 3: tiene_turnos + cupo_maximo + tiene_lista_espera (lista de espera requiere cupo configurado)
- [x] Paso 4: ícono, color, orden + resumen de todos los datos antes de confirmar
- [x] Programa creado en estado `BORRADOR`
- [x] Estado cambiable desde el listado con SweetAlert2 (BORRADOR→ACTIVO, ACTIVO→SUSPENDIDO, SUSPENDIDO→ACTIVO)
- [x] Wizard de edición reutiliza la misma estructura de pasos
- [x] Programas `INACTIVO` no pueden editarse
- [x] Migración de datos: `activo=True` → `ACTIVO`, `activo=False` → `INACTIVO`

## Dependencias

- Depende de: US-004 (Secretarías/Subsecretarías), US-011 (grupo `programaConfigurar`)
- Prerequisito de: US-006 (motor de flujos — para activar programas BORRADOR), US-012 (inscripción a programas)

## Decisiones tecnicas

- **Estado reemplaza `activo`**: 4 estados permiten un ciclo de vida más expresivo. `BORRADOR` es el estado inicial — el programa no acepta inscripciones hasta estar `ACTIVO`. `ACTIVO` requirirá un flujo configurado (US-006) para ser completamente operativo.
- **Wizard con sesión, sin django-formtools**: `django-formtools` no está instalado. Se implementó con estado en sesión (`request.session`) siguiendo el mismo patrón del wizard de admisión existente en `legajos`.
- **Filtrado AJAX de subsecretarías**: endpoint `GET /core/ajax/load-subsecretarias/?secretaria=<id>` devuelve JSON. Sigue el mismo patrón de `load_municipios` y `load_localidades` en `core`.
- **`TipoPrograma` conservada**: la inner class TextChoices no se eliminó para no romper 3 archivos de test existentes que la referencian. El campo `tipo` pasa a `CharField` libre (sin choices, sin unique).
- **URL dinámica en JS**: la URL del form de cambio de estado se construye con `data-url-template` renderizado por Django (`{% url ... 0 %}`), evitando hardcoding en JavaScript.

## Limitaciones conocidas / deuda tecnica

- La transición `BORRADOR → ACTIVO` debería validar que el programa tenga un flujo configurado (US-006). Por ahora solo valida que tenga `naturaleza` configurada — validación mínima.
- Los programas `INACTIVO` no tienen un path de reactivación desde la UI (solo se pueden activar programas en BORRADOR o SUSPENDIDO desde el listado).
- El wizard no incluye asignación de coordinadores ni configuración de roles dentro del flujo — eso corresponde a US-006/US-007.

## Cambios respecto a version anterior

La v1.0 documentaba los programas como entidades simples con un campo `activo` booleano y `tipo` como enum hardcodeado. Esta versión formaliza el ciclo de vida del programa con 4 estados, agrega jerarquía organizacional completa, naturaleza, capacidades y provee un wizard de configuración de 4 pasos accesible desde el módulo de Configuración.
