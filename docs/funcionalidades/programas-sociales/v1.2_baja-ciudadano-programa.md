## Descripcion

Permite dar de baja formalmente a un ciudadano de un programa persistente desde el panel operativo del programa. La baja registra el motivo, cancela los turnos pendientes vinculados al programa y cancela el flujo activo si lo hay. El registro queda trazado con estado `DADO_DE_BAJA`, sin eliminación de datos.

## User Story

```
Como operador con acceso al panel del programa quiero dar de baja a un ciudadano de un programa persistente registrando el motivo para cerrar su caso formalmente y liberar los recursos asociados.
```

## Criterios de aceptacion

- [x] El botón "Dar de Baja" aparece en la tabla de Acompañamientos solo para inscripciones en estado ACTIVO, EN_SEGUIMIENTO o PENDIENTE
- [x] El modal SweetAlert2 solicita un motivo obligatorio y bloquea el envío si está vacío
- [x] Al confirmar, la inscripción pasa a estado `DADO_DE_BAJA` con `motivo_cierre` y `fecha_cierre`
- [x] Los turnos PENDIENTE y CONFIRMADO vinculados al programa se cancelan con `CANCELADO_SISTEMA`
- [x] Si existe un flujo activo (`InstanciaFlujo.estado = ACTIVA`), pasa a `CANCELADA` con log de motivo
- [x] Si el flujo no existe o ya está cerrado, la operación no falla
- [x] El badge del estado `DADO_DE_BAJA` se muestra en `badge-danger` en la tabla
- [x] El contador de "Bajas" aparece en el header del tab Acompañamientos

## Decisiones tecnicas

- **`DADO_DE_BAJA` como estado propio:** Se decidió no reutilizar `CERRADO` para distinguir semánticamente entre cierre por completitud del programa y baja por decisión operativa. Permite filtrar bajas independientemente.
- **No se elimina el registro:** El `InscripcionPrograma` queda en la base con estado `DADO_DE_BAJA`. El `unique_together` sigue vigente; una eventual re-inscripción deberá manejar ese caso en su propio service.
- **Cancelación de turnos por `contexto_tipo/contexto_id`:** `TurnoCiudadano` usa generic FK liviana. El service filtra por `contexto_tipo=PROGRAMA` y `contexto_id=programa_id` para no cancelar turnos de otras inscripciones del mismo ciudadano.
- **Flujo cancelado con log:** Se prefirió crear un `InstanciaLog` con `nodo_hasta='BAJA'` para mantener trazabilidad de por qué el flujo se canceló, en lugar de un simple update silencioso.

## Limitaciones conocidas / deuda tecnica

- El botón de baja no requiere permiso específico más allá de `@login_required`. Si en el futuro se implementa `programaOperar` como permiso requerido para baja, deberá añadirse una guarda en la vista.
- La re-inscripción al mismo programa desde `DADO_DE_BAJA` no está implementada; el `unique_together` bloqueará la creación de una nueva inscripción sin lógica de reactivación.
