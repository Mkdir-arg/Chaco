# Mejora de Logging — v1.0

> Version: v1.0
> Fecha: 2026-03-05
> Tipo: EVOLUTIVO
> App/Modulo: `core`, `config`

---

## Descripcion

Mejora del sistema de observabilidad del servidor. Antes los logs iban solo a archivos y `docker logs` no mostraba nada útil. Se agrega salida a stdout (console), un middleware que loguea cada request HTTP con contexto completo, y un formato detallado en nginx con tiempos de respuesta.

## User Story

```
Como desarrollador del equipo, quiero tener logs detallados y en tiempo real
de errores del backend, requests HTTP y performance de nginx
para diagnosticar problemas en producción sin necesidad de acceso directo al servidor.
```

## Criterios de aceptacion

- [x] `docker logs nodo-web` muestra logs en tiempo real
- [x] Los errores 500 muestran traceback completo
- [x] Errores 4xx quedan registrados (no solo 500s)
- [x] Cada request loguea: método, URL, usuario, IP, status y duración en ms
- [x] El access log de nginx incluye: IP, método, URL, status, bytes, rt y upt

## Archivos involucrados

| Archivo | Rol | Cambios en esta version |
|---------|-----|------------------------|
| `core/middleware.py` | Middleware HTTP | Nuevo. `RequestLoggingMiddleware` loguea cada request |
| `config/settings.py` | Configuracion | Agregado `console` handler, `core.middleware` en MIDDLEWARE, `django.request` a WARNING |
| `nginx.conf` | Servidor web | Agregado `log_format detailed` con `$request_time` y `$upstream_response_time` |

## Migraciones generadas

Ninguna.

## Dependencias

- Depende de: sistema de logging existente (`DailyFileHandler`, `JSONDataFormatter` en `core/utils.py`)
- Usada por: toda la aplicacion (middleware global)

## Decisiones tecnicas

- **Console handler en lugar de herramienta externa**: se prefirió la solución nativa de Python/Django antes que Sentry u otro servicio externo, manteniendo cero dependencias nuevas.
- **Middleware al final de la cadena**: posicionado después de `AuthenticationMiddleware` para tener `request.user` disponible. Al ser el último middleware, mide la duración total real del request incluyendo todos los middlewares anteriores.
- **`django.request` en WARNING**: captura tanto 4xx (WARNING) como 5xx (ERROR), que antes solo capturaba 5xx.
- **`$upstream_response_time` en nginx**: puede mostrar `-` para archivos estáticos servidos directamente por nginx — comportamiento esperado.

## Limitaciones conocidas / deuda tecnica

- No captura errores de JavaScript del frontend (Propuesta C, descartada en este sprint).
- No hay integración con servicios externos de monitoreo (Sentry, Datadog).
- El middleware loguea todos los requests incluyendo `/health/` y archivos estáticos que Django sirve — se puede filtrar en el futuro si genera ruido.
