## Descripcion

Flujo de creacion de ciudadano via consulta a RENAPER. El operador ingresa un DNI, el sistema consulta la API de RENAPER, pre-completa un formulario con los datos obtenidos, y el operador confirma o ajusta los datos antes de guardar el ciudadano en el sistema.

## User Story

```
Como operador del sistema quiero consultar RENAPER por DNI para pre-completar los datos del ciudadano y evitar carga manual de informacion
```

## Criterios de aceptacion

- [ ] Dado un DNI valido, cuando se consulta RENAPER, entonces el formulario se pre-completa con los datos retornados
- [ ] Si RENAPER retorna que la persona figura como fallecida, entonces se muestra un mensaje de error y no se permite continuar
- [ ] Si RENAPER no encuentra datos, entonces se muestra el error correspondiente
- [ ] El operador puede modificar los datos pre-completados antes de confirmar
- [ ] Al confirmar, el ciudadano queda guardado y se invalida el cache del dashboard
- [ ] Solo usuarios autenticados pueden acceder al flujo

## Dependencias

- Depende de: API RENAPER (`RENAPER_API_URL`), sesion Django para pasar datos entre vistas
- Usada por: flujo de alta de ciudadanos, legajos

## Decisiones tecnicas

- Los datos de RENAPER se pasan entre vistas via sesion Django (`request.session['datos_renaper']`), no via URL ni querystring, para evitar exposicion de datos personales en logs
- El campo DNI es `readonly` en el formulario de confirmacion para evitar modificacion

## Limitaciones conocidas / deuda tecnica

- Los campos `provincia`, `municipio`, `localidad` estan en el modelo y el form pero no se renderizan en el template de confirmacion
