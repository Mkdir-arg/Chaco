# Refactor DX Interno — Slice 7: `legajos` fachada pura de views

> Version: v1.6  
> Fecha: 2026-03-13  
> Estado: Implementado

## Objetivo

Completar la modularización física de `legajos/views.py` para que el archivo deje de contener implementación real y funcione solo como fachada de compatibilidad.

## Alcance

- Extracción del bloque operativo/institucional y de actividades a `legajos/views_operativa.py`
- Conservación de `legajos/views.py` como punto de reexportación
- Cero cambios en modelos, migraciones, templates o URLs

## Cambios implementados

- Se creó `legajos/views_operativa.py`
- Se movieron a ese módulo las vistas y funciones:
  - `LegajoInstitucionalListView`
  - `LegajoInstitucionalDetailView`
  - `LegajoInstitucionalCreateView`
  - `LegajoInstitucionalUpdateView`
  - `InstitucionListView`
  - `InstitucionCreateView`
  - `InstitucionUpdateView`
  - `InstitucionDeleteView`
  - `marcar_etapa_plan`
  - `actividades_por_institucion`
  - `InscribirActividadView`
  - `ActividadesInscritoListView`
- `legajos/views.py` quedó como fachada que reexporta:
  - ciudadanía/admisión
  - clínica
  - contactos/red
  - operativa

## Impacto en DX

- Baja el tamaño cognitivo del módulo `legajos`
- Facilita navegación y code review por subdominio
- Permite seguir refactorizando sin cambiar imports legacy ni rutas

## Riesgos

- Bajo: el cambio es estructural y mantiene compatibilidad por reexportación
- Persisten riesgos globales del módulo `legajos` por mezcla de dominios en una sola app

## Siguiente etapa sugerida

- Atacar `conversaciones` con el mismo patrón de selectors/services/views por dominio
- Definir una política explícita y única de responsables de legajo
