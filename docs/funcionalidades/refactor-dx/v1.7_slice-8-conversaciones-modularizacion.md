# Refactor DX Interno — Slice 8: `conversaciones` modularización

> Version: v1.7  
> Fecha: 2026-03-13  
> Estado: Implementado

## Objetivo

Reducir el acoplamiento del módulo `conversaciones` separando responsabilidades entre vistas públicas, vistas de backoffice, selectors, services y validación de payloads, sin cambiar las URLs ni el contrato actual del chat.

## Alcance

- Separación física de `conversaciones/views.py` en:
  - `views_public.py`
  - `views_backoffice.py`
- Nuevos módulos:
  - `forms_chat.py`
  - `selectors_conversaciones.py`
  - `services_chat.py`
- Tests base del slice en `conversaciones/tests/test_chat_services.py`
- Sin cambios en `api_urls.py`, templates, modelos o migraciones

## Cambios implementados

- Los payloads JSON del chat ahora se validan con forms livianos antes de llegar a la lógica de negocio
- La bandeja, métricas y detalle de conversación pasaron a apoyarse en selectors explícitos
- La creación de conversación, mensajes, asignación, cierre, cola y evaluación quedaron encapsuladas en `services_chat.py`
- `views.py` quedó como fachada compatible para no romper imports ni URLs existentes

## Impacto en DX

- `conversaciones/views.py` dejó de ser un archivo monolítico difícil de navegar
- Se redujo la duplicación de queries y de parsing manual de `request.body`
- El módulo ahora tiene puntos claros para testear lectura, validación y workflow

## Riesgos y deuda residual

- Se mantuvieron `@csrf_exempt` legacy en endpoints donde removerlos exigiría coordinar cambios de frontend
- El chat sigue mezclando polling HTTP y notificaciones realtime parciales
- La capa realtime todavía depende de best-effort y imports protegidos de `channels`

## Siguiente etapa sugerida

- Definir un corte específico para endurecer CSRF y transporte del chat sin romper UX
- Revisar si `conversaciones/api_urls.py` y `api_views.py` también conviene alinearlos con selectors/services
