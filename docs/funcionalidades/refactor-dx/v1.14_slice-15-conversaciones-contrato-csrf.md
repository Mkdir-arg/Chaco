# Refactor DX Interno — Slice 15

> Fecha: 2026-03-13
> App principal: `conversaciones`
> Estado: Implementado

## User Story

Como equipo de desarrollo quiero corregir el contrato real frontend/backend de `conversaciones` y endurecer CSRF en sus endpoints JSON para reducir fragilidad operativa sin cambiar la UX actual del chat.

## Alcance

- Alinear URLs consumidas por `chat_ciudadano`, `detalle` y `lista` con rutas renderizadas por Django.
- Corregir la evaluación del ciudadano para que vuelva a una vista pública y no choque con permisos de backoffice.
- Retirar `@csrf_exempt` de los POST JSON principales del módulo.
- Agregar tests de contrato y enforcement de CSRF.

## Cambios implementados

- `conversaciones/views_public.py`
  - `chat_ciudadano` ahora emite cookie CSRF.
  - `consultar_renaper`, `iniciar_conversacion` y `enviar_mensaje_ciudadano` dejan de usar `@csrf_exempt`.
  - `evaluar_conversacion` pasa a la superficie pública real del chat.
- `conversaciones/views_backoffice.py`
  - `enviar_mensaje_operador` deja de usar `@csrf_exempt`.
- `conversaciones/views.py`
  - La fachada compatible reexporta la evaluación pública.
- Templates:
  - `chat_ciudadano.html`, `detalle.html` y `lista.html` ya no dependen de rutas hardcodeadas.
  - Los fetches JSON envían cabecera `X-CSRFToken`.
- Tests:
  - Se agregan casos para emisión de cookie CSRF, bloqueo sin token y contrato JSON de inicio/evaluación/respuesta.

## Impacto funcional

- Se corrige una inconsistencia real: el ciudadano vuelve a poder evaluar su conversación usando la URL documentada del módulo.
- No se modifican modelos, migraciones ni UX visible del flujo.

## Riesgos residuales

- `conversaciones` todavía mezcla polling HTTP con WebSockets y no todo el módulo tiene cobertura de tests de contrato.
- La API auxiliar y el realtime de lista siguen siendo áreas sensibles para un corte posterior.
