# Refactor DX Interno — Slice 16

> Fecha: 2026-03-13
> App principal: `conversaciones`
> Estado: Implementado

## User Story

Como equipo de desarrollo quiero estabilizar el runtime de la lista en vivo de `conversaciones` para evitar doble suscripción WebSocket y paths hardcodeados, manteniendo intacta la UX actual.

## Alcance

- Eliminar la doble carga de `conversaciones_lista_ws.js` en la pantalla de lista.
- Hacer idempotente la inicialización del runtime WebSocket.
- Mover URLs operativas de detalle/cierre/API al DOM renderizado por Django.
- Agregar cobertura de contrato del template para esos atributos.

## Cambios implementados

- `conversaciones/templates/conversaciones/lista.html`
  - Se agregan `data-*` con rutas de detalle, cierre, API y path de WebSocket.
  - Se elimina la carga local duplicada del script, manteniendo la carga global desde `base.html`.
- `static/custom/js/conversaciones_lista_ws.js`
  - Agrega guard de inicialización para evitar doble bootstrap.
  - Lee configuración desde `#conversaciones-lista-app`.
  - Deja de construir URLs de detalle/cierre/API con strings hardcodeados.
- `conversaciones/tests/test_chat_services.py`
  - Se agrega test que valida el contrato renderizado de la pantalla de lista.

## Impacto funcional

- No cambia la UX visible del backoffice.
- Se reduce el riesgo de múltiples conexiones WebSocket en la misma pantalla y se prepara mejor el módulo para la migración de namespaces.

## Riesgos residuales

- El chat ciudadano sigue usando polling HTTP para recuperar mensajes.
- La convergencia completa de realtime entre polling y WebSockets queda para un slice posterior.
