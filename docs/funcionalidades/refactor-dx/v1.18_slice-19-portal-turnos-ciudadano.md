# Refactor DX Interno — Slice 19: `portal` turnos ciudadano

> Versión: v1.18
> Fecha: 2026-03-13
> Estado: Implementado

## Objetivo

Extraer el subdominio de turnos del portal ciudadano fuera de `portal/views_ciudadano.py` para separar lectura, validación y persistencia, manteniendo intacto el flujo visible para el ciudadano.

## Cambios aplicados

- Se creó `portal/views_ciudadano_turnos.py` como módulo dedicado al flujo ciudadano de turnos.
- Se creó `portal/selectors_turnos_ciudadano.py` para ownership y lecturas reutilizables del dominio.
- Se creó `portal/services_turnos_ciudadano.py` para encapsular reserva y cancelación de turnos.
- Se agregó `CiudadanoConfirmarTurnoForm` en `portal/forms.py` para validar fecha, horarios y motivo.
- Se adaptó `portal/templates/portal/ciudadano/turno_confirmar.html` para usar el form explícito y mostrar errores.
- Se agregaron tests en `portal/tests/test_ciudadano_turnos.py`.

## Impacto funcional

- No hay cambios de modelo ni migraciones.
- No cambian URLs ni nombres públicos del portal.
- Se mantiene el flujo existente: elegir recurso, ver calendario, elegir slot, confirmar y cancelar.
- La disponibilidad y el cupo pasan a validarse desde una capa reutilizable en lugar de quedar inline en la view monolítica.

## Beneficio en DX

- `portal/views_ciudadano.py` baja de tamaño y deja de mezclar turnos con auth, perfil y consultas.
- La reserva/cancelación queda aislada y testeable.
- La confirmación deja de depender de parsing manual de `POST`.
- El subdominio queda alineado con el patrón general del refactor: forms livianos, selectors de lectura y services de orquestación.

## Riesgos residuales

- `portal/views_ciudadano.py` todavía concentra autenticación, registro y perfil del ciudadano.
- El flujo sigue dependiendo de `turnos_utils.py`, que permanece como pieza compartida del calendario y slots.
