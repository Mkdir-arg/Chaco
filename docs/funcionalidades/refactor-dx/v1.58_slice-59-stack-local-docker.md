# Refactor DX Interno - Slice 59: stack local Docker de un solo comando

> Version: v1.58
> Fecha: 2026-04-03
> Estado: Implementado

## Objetivo

Reducir el tiempo hasta entorno usable en desarrollo local reemplazando el stack híbrido por un flujo mínimo y determinista con `docker compose up`.

## Alcance

- `docker-compose.yml` como entrypoint local por defecto
- Un solo servicio `app` para HTTP + WebSocket vía ASGI
- Bootstrap idempotente y liviano en `docker-entrypoint.sh`
- Corrección de `crear_programas` para el schema actual
- Documentación explícita del entorno local

## Cambios implementados

- Se retiró `nginx` del flujo local recomendado
- Se eliminó la separación local `sedronar-http` / `sedronar-ws`
- El contenedor de aplicación ya no reinstala dependencias en runtime
- El startup diario ya no ejecuta `sleep`, `collectstatic`, `load_initial_data` ni `setup_system`
- Se dejó un bootstrap mínimo con `migrate`, `crear_superadmin`, `setup_grupos` y `crear_programas`
- `crear_programas` ahora usa `estado` y `naturaleza`, no el campo legacy `activo`
- Se agregó `.env.local.example` y una guía de uso en `docs/team/entorno-local.md`

## Impacto en DX

- El contrato local queda reducido a un solo comando
- Baja la fragilidad del arranque porque los seeds pesados dejan de ser obligatorios
- Un fallo en setup opcional ya no debe bloquear la disponibilidad del backend
- HTTP y WebSocket comparten el mismo proceso ASGI y el mismo puerto local

## Riesgos residuales

- El runtime local usa `runserver` para tener autoreload; si aparece alguna incompatibilidad puntual con Channels en una pantalla específica, el fallback sigue siendo `APP_RUNTIME=daphne`
- `load_initial_data` y `setup_system` siguen existiendo para bootstrap manual/extraordinario y deben mantenerse fuera del startup diario

## Siguiente etapa sugerida

- Si el equipo necesita recarga automática ASGI, evaluar una capa de dev server específica sin volver a dividir el runtime local en múltiples servicios
