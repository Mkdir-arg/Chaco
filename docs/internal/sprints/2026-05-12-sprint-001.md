## 3. Análisis funcional del sprint

### REQ-01 — Registro de ciudadano

**Análisis completo:** [AF-001 — Registro de ciudadano](../analisis/001-registro-de-ciudadano.md)

**Qué se analizó:**
Se revisaron los modelos `Ciudadano` en `legajos/models.py`, el flujo de consulta a RENAPER en `legajos/services/consulta_renaper.py`, los formularios existentes en `legajos/forms/ciudadanos.py` y los templates de registro. Se identificó que el sistema ya tiene la integración con RENAPER implementada pero sin validación de duplicados consistente.

**Qué se definió:**
- El sistema verifica duplicados por DNI antes de crear el legajo
- RENAPER se consulta automáticamente al ingresar el DNI; si falla, habilita el formulario manual
- Los legajos sin DNI quedan en estado "pendiente de verificación" y no pueden derivarse
- Los datos de RENAPER no son editables por el operador en el momento del registro

**Requerimientos derivados:**

| ID | Requerimiento | Prioridad | Estado |
|---|---|---|---|
| RF-001-01 | Verificar DNI duplicado antes de crear legajo | Alta | ✅ Completado |
| RF-001-02 | Consultar RENAPER y autocompletar datos | Alta | ✅ Completado |
| RF-001-03 | Permitir registro manual si RENAPER no responde | Alta | ✅ Completado |
| RF-001-04 | Permitir registro sin DNI con estado "pendiente de verificación" | Media | ✅ Completado |
| RF-001-05 | Validar formato de DNI en el formulario | Alta | ✅ Completado |
| RF-001-06 | Registrar auditoría de creación (quién y cuándo) | Alta | ✅ Completado |
| RF-001-07 | Adjuntar foto al momento del registro | Baja | ⏸ Postergado al Sprint 002 |

**Criterios de aceptación:**
- [x] Dado un operador con permisos, cuando ingresa un DNI existente, entonces el sistema muestra el legajo existente y no crea uno nuevo.
- [x] Dado un operador con permisos, cuando ingresa un DNI nuevo y RENAPER responde, entonces el sistema autocompleta nombre, apellido y fecha de nacimiento.
- [x] Dado un operador con permisos, cuando RENAPER no responde, entonces el sistema habilita el formulario manual sin bloquear el flujo.
- [x] Dado un operador con permisos, cuando registra un ciudadano sin DNI, entonces el legajo queda en estado "pendiente de verificación".
- [x] Dado cualquier usuario sin permiso `legajos.add_ciudadano`, cuando intenta acceder al formulario, entonces recibe un error 403.

**Fuera de alcance en este sprint:**
- Adjuntar foto (postergado)
- Edición de datos post-registro
- Carga masiva por CSV

### REQ-02 — Derivación de ciudadano a programa

**Análisis completo:** [AF-002 — Derivación de ciudadano a programa](../analisis/002-derivacion-de-ciudadano-a-programa.md)

**Qué se analizó:**
Se revisaron los modelos de derivación en `legajos/models.py`, los servicios en `legajos/services/derivaciones_programa.py`, las vistas en `legajos/views/derivacion_programa.py` y los templates del flujo. Se identificó que el modelo de derivación existe pero el flujo de aceptación/rechazo por parte del receptor no está implementado.

**Qué se definió:**
- El flujo de derivación tiene tres estados: Pendiente → Aceptada / Rechazada / Vencida / Anulada
- La notificación al receptor usa el sistema de alertas interno del módulo `conversaciones`
- El plazo de vencimiento es configurable por institución
- Una derivación aceptada no puede anularse, solo cerrarse por egreso del programa

**Requerimientos derivados:**

| ID | Requerimiento | Prioridad | Estado |
|---|---|---|---|
| RF-002-01 | Iniciar derivación desde el legajo | Alta | 🔄 En progreso |
| RF-002-02 | Formulario con programa, dispositivo, motivo y observaciones | Alta | 🔄 En progreso |
| RF-002-03 | Notificar al receptor al crear la derivación | Alta | ⏳ Pendiente |
| RF-002-04 | Aceptar o rechazar con motivo obligatorio | Alta | ⏳ Pendiente |
| RF-002-05 | Bloquear derivación duplicada al mismo programa | Alta | ⏳ Pendiente |
| RF-002-06 | Historial de derivaciones en el legajo | Alta | ⏳ Pendiente |
| RF-002-07 | Anular derivación pendiente (admin) | Media | ⏳ Pendiente |
| RF-002-08 | Marcar derivación como vencida por plazo | Media | ⏳ Pendiente |

**Criterios de aceptación:**
- [ ] Dado un operador con permisos, cuando inicia una derivación, entonces el sistema registra la derivación con estado "Pendiente" y notifica al receptor.
- [ ] Dado un operador receptor, cuando acepta una derivación, entonces el ciudadano queda inscripto en el programa y el estado cambia a "Aceptada".
- [ ] Dado un operador receptor, cuando rechaza una derivación sin ingresar motivo, entonces el sistema no permite enviar el formulario.
- [ ] Dado un ciudadano con inscripción activa en un programa, cuando un operador intenta derivarlo al mismo programa, entonces el sistema muestra un aviso y bloquea la acción.
- [ ] Dado un administrador, cuando anula una derivación en estado "Pendiente", entonces el estado cambia a "Anulada" y se notifica a ambos operadores.

**Fuera de alcance en este sprint:**
- Derivaciones inter-institucionales
- Flujo de egreso del programa
- Reportes de derivaciones
